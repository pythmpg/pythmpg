# Project Setup and Documentation Notes

This file describes what lives in the repo, what is autogenerated, what is hand-written, and how to build the documentation.


## Purpose of each non-code file

pyproject.toml is the source for project metadata, dependencies (numpy), optional dependency groups (docs), the build backend (hatchling), and tool configuration (ruff).

pre-commit-config.yaml connects Ruff to git's pre-commit hook so every commit auto-formats and lints Python files. It runs ruff with --fix and then ruff-format.

readthedocs.yaml tells the Read the Docs hosting service how to build the documentation site (Python version, install command, Sphinx config location) if the project is published there.

.gitignore excludes certain folders and files from version control. This is a template from the internet. You can edit it to add exceptions if you want to track something that is currently ignored, or vice versa.

README.md is the project description shown on the repo home page.

docs/source/conf.py is the Sphinx configuration file.

docs/source/index.rst is the documentation home page and contains the top-level toctree (table of contents tree).

docs/source/api.rst is the API reference page and lists the modules that autosummary should document.

docs/Makefile and docs/make.bat were created by sphinx-quickstart (see the Sphinx documentation for details). They are just shortcutrs for running sphinx-build with the correct arguments on different platforms. You can ignore them and run sphinx-build directly if you prefer.

docs/source/generated/ is populated on each build by autosummary, one .rst file per submodule. It is gitignored.

I've added docs/source/_templates/autosummary/module.rst to override autosummary's default template. This allows each module to be recursively documented with its own page instead of all modules being documented on a single page. 

docs/build/ is the rendered HTML output automatically generated and is gitignored.


## What Sphinx autogenerates vs. what you write

You only hand-write three kinds of files: the docstrings inside pythmpg/*.py, the small set of .rst files (index.rst, api.rst, and whichever .rst files you create), and conf.py. Everything else is regenerated on each build.

Sphinx then does the following automatically. The napoleon extension parses the NumPy-style docstrings. The autodoc extension renders those parsed docstrings into HTML. The autosummary extension generates the per-module .rst files in docs/source/generated/. The sphinx-build command produces the HTML pages in docs/build/. The Furo theme reads the table-of-contents in index.rst and api.rst to build the sidebar navigation.

## Python environment

The main dependency is numpy, declared in pyproject.toml under [project] dependencies.

The documentation dependencies are sphinx, furo (the theme), and sphinx-autobuild (the live-reload server). These are in pyproject.toml under [project.optional-dependencies] docs.

To install everything into your active Python environment, run `pip install -e ".[docs]"` from the project root. The -e flag makes it an editable install so changes to pythmpg/ are picked up by Sphinx without reinstalling.

Pre-commit is optional but recommended. To enable it, run pip install pre-commit followed by pre-commit install. Once installed, Ruff runs on every git commit.


## Building the documentation

For a one-shot build, run sphinx-build -M html docs/source docs/build and then open docs/build/html/index.html in your browser.

For live-reload development, run sphinx-autobuild docs/source docs/build/html. This watches your source files and rebuilds the docs while reloading the browser whenever you save. Visit http://127.0.0.1:8000 in your browser. Saving any .py or .rst file triggers an automatic reload.

For a clean rebuild when something looks stale or you are getting miscellaneous errors, delete docs/source/generated/ and docs/build/ and run sphinx-build again.
